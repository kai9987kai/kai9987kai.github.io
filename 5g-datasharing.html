    Connectivity token schema

    A realistic grant should be device-bound, short-lived, policy-minimal, revocable, and auditable. The schema below is intentionally split between public claims, private encrypted claims, and off-chain references.

    Schema upgrades

    • cnf / holder binding: token can only be used by the Guest UE key or eUICC-bound proof.
    • status pointer: revocation is checked through a compact status-list or permissioned registry index.
    • policy minimisation: expose slice, QoS, quota, and TTL claims, not unrelated identity data.
    • privacy boundary: put phone number, billing identity, and full subscriber state in an encrypted/off-chain vault.
    • replay control: nonce, audience, API endpoint, and proof-of-possession are verified for each activation.
    Use the ledger for evidence and ordering, not as a database for subscriber records.

    Abstract

    We propose a delegated connectivity-token architecture that allows a sponsoring Host UE or enterprise authority to grant a Guest UE short-lived access to a constrained 5G service profile. The token is bound to the Guest UE holder key, scoped by slice, DNN, QoS, geography, quota, and expiry, and validated through a privacy-preserving status registry. A permissioned ledger provides tamper-evident issuance and revocation ordering, while subscriber identity and billing data remain off-chain. The 5G core enforces the grant through NEF exposure, AMF registration, AUSF/UDM authentication checks, NSSF/PCF/SMF policy selection, and UPF metering.

    Contributions

    • A connectivity grant primitive for temporary, sponsor-backed mobile access.
    • A privacy-preserving audit model that stores commitments and status proofs rather than PII.
    • A 5G-core enforcement path mapping token claims to slice, QoS, quota, DNN, and PDU-session policy.
    • A revocation model combining short TTLs, status-list checks, and signed revocation events.
    • An evaluation plan covering attach latency, revocation propagation, policy correctness, fraud resistance, and ledger throughput.

    Architecture

    The Host UE or sponsoring application requests a grant from the Token Manager. The Policy/Risk Engine checks consent, host allowance, device posture, subscriber constraints, enterprise rules, and roaming constraints. The Token Manager creates a signed token and anchors a hash commitment plus status index to a permissioned ledger. Delivery to the Guest UE may occur through an LPA/eUICC workflow, a wallet, or an enterprise device-management channel depending on deployment. When the Guest UE requests attachment or service exposure, the NEF/API Gateway validates the holder proof and status pointer before the 5G core installs bounded policy.

    Protocol design

    1. Consent and sponsor request: host signs grant intent with scope and expiry.
    2. Risk and eligibility: policy engine checks quota, fraud signals, posture, roaming, and regulatory constraints.
    3. Token mint: issuer signs a VC/JWT-style grant with holder binding and status-list entry.
    4. Commitment anchor: ledger stores token hash, issuer DID/key ID, expiry bucket, status index, and event sequence.
    5. Guest activation: Guest UE proves possession of holder key and submits token to exposure/control plane.
    6. Core enforcement: PCF/NSSF/SMF translate grant claims into slice, QoS, quota, and session rules.
    7. Continuous assurance: UPF metering and NWDAF/SIEM signals can downgrade, revoke, or rate-limit the grant.

    Evaluation plan

    Metrics

    • Token issuance latency
    • Attach/session setup overhead
    • Status lookup latency and cache hit rate
    • Revocation propagation time
    • False allow / false deny rate
    • UPF quota accuracy

    Experiments

    • Normal guest onboarding
    • Replay attack with stolen token
    • Expired token activation
    • Host-triggered revocation
    • Congested ledger/status registry
    • Multi-slice enterprise scenario

    Limitations and open questions

    This is a research architecture, not a deployable standards profile. Real deployments would need operator integration, lawful intercept and emergency-service treatment, roaming agreements, billing settlement, secure eUICC/LPA support, key recovery UX, regulatory review, and conformance testing. The ledger should be optional: a highly available signed transparency log or verifiable data registry may be enough where blockchain governance adds cost without benefit.
