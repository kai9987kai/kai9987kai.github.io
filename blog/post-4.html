Three.js lab notes: scenes, cameras, polish and performance.

A 3D web dev-log about taking rough Three.js experiments and upgrading them into cleaner, smoother and more portfolio-ready interactive scenes.

✍️ Kai Piper 📅 09 Jun 2026 ⏱️ 5 min read 🧊 3D Web

ViewportWebGL

Three.js is one of the best ways to make a website feel alive. A normal web page can show text and images, but a 3D scene can let visitors orbit, explore, click, zoom, move through space and feel like the project has its own little world.

For kai9987kai, Three.js projects are a good fit because they combine creative coding, design, interaction and technical problem solving. The challenge is making sure a scene feels polished instead of just being a spinning cube on a black background.

A good 3D web scene should feel like a small interactive world, not just a model dropped onto a page.

Scene design

The scene is the foundation. Before adding complex effects, the project needs a clear visual idea. Is it a product viewer, a sci-fi interface, a portfolio hero, a 3D game space, a data visualisation, or a playful interactive object?

Once the purpose is clear, the scene can be built in layers: background, lighting, main object, supporting objects, camera position, interaction hints and UI. This makes the project easier to improve because every part has a job.

Design goal:

Build 3D scenes with a clear subject, useful camera framing, readable lighting and simple interaction that supports the idea instead of distracting from it.

Camera movement

Camera movement can make or break a 3D web project. A scene can look impressive, but if the camera starts too close, rotates too fast, clips through objects, or gets lost in empty space, the experience starts to feel messy.

The strongest approach is to treat the camera like part of the interface. The default view should show the subject clearly. Orbit controls should feel smooth. Zoom should have limits. Important objects should guide the eye instead of leaving the visitor to search around.

Framing Start with the main object visible, centred and easy to understand.
Controls Use smooth orbit, sensible zoom limits and simple pointer interactions.
Lighting Use key, fill and rim-style lighting so the shape reads clearly.
Performance Keep geometry, textures and effects balanced so the page stays responsive.

Visual polish

Polish is where a Three.js experiment starts to feel portfolio-ready. This can include better materials, softer shadows, post-processing, environment lighting, animated reveal transitions, hover states, labels and subtle UI panels around the 3D scene.

Small details matter. A loading state makes the page feel intentional. A fallback message helps people on weaker devices. A “reset camera” button prevents users from getting lost. Labels or callouts make complex models easier to understand.

  • Better materials: use roughness, metalness, normal maps or matcap-style looks where useful.
  • Scene atmosphere: use gradients, fog, particles or subtle grids to create depth.
  • Helpful UI: add labels, controls and status text instead of hiding everything.
  • Responsive design: make the canvas adapt properly to desktop and mobile screens.
  • Performance checks: reduce heavy meshes, large textures and unnecessary render work.

Next version

The next version should focus on turning 3D experiments into complete mini experiences. That means every scene should have a purpose, a clear user journey, a polished visual style, and a short explanation of what the viewer is meant to notice or do.

Good upgrades would include a loading screen, model compression, mobile-friendly controls, pointer hover effects, camera bookmarks, performance options, labelled hotspots and a short case-study section under each scene explaining how it was built.

Portfolio angle:

A polished Three.js project can show creative coding, 3D design, JavaScript skill, interaction design, performance thinking and visual presentation all in one piece.
